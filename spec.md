# 可视化 Diff 审阅 Spec

## 背景
CodeShell 当前已经有权限确认机制：写文件工具和命令类工具在默认权限模式下可以触发人工确认。但确认内容主要是“是否允许执行某个工具/命令”，用户在批准前无法直观看到文件修改前后的差异。对于 `EditFile`、`WriteFile` 以及可能修改文件的任意 `Bash` 命令，这会增加误改、误删和隐藏副作用的风险。

本功能新增“可视化 diff 审阅”流程：文件修改在真正应用到工作区之前，先生成修改前后对比，展示给用户确认；用户批准后才应用，拒绝则不改变工作区。

## 目标
- 所有经 CodeShell 工具路径发起的文件修改，都先进入 diff 审阅流程。
- `EditFile` 和 `WriteFile` 在写入前展示 proposed diff，用户确认后再写入。
- `Bash` 命令在可能修改文件时，通过预执行/快照机制生成文件变更 diff，用户确认后再应用到真实工作区。
- 用户可以在界面中查看新增、修改、删除文件的差异摘要和具体 diff。
- 用户拒绝后，真实工作区保持不变。
- 用户批准后，真实工作区应用与 diff 一致的文件变化。
- 审阅流程与现有权限模式协同，不破坏 Plan Mode、默认权限、acceptEdits、bypassPermissions 的基本语义。

## 功能需求
- F1: 当 `EditFile` 准备修改已有文件时，系统必须展示修改前后 diff，并等待用户批准后才写入真实文件。
- F2: 当 `WriteFile` 准备创建或覆盖文件时，系统必须展示新增或覆盖 diff，并等待用户批准后才写入真实文件。
- F3: 当 `Bash` 命令可能修改工作区文件时，系统必须先在隔离预览环境中执行，收集文件变更 diff，并等待用户批准后才把文件变更应用到真实工作区。
- F4: diff 审阅必须覆盖新增文件、修改文件、删除文件和空文件变化。
- F5: diff 审阅必须展示文件级摘要，包括新增、修改、删除文件数量和路径列表。
- F6: diff 审阅必须展示具体文本 diff；二进制文件或无法安全展示的文件应以摘要形式提示。
- F7: 用户批准 diff 后，系统应用对应文件变化，并把工具结果返回给 agent。
- F8: 用户拒绝 diff 后，系统不应用文件变化，并把拒绝结果返回给 agent。
- F9: 如果预览执行失败，系统不修改真实工作区，并把失败原因返回给 agent。
- F10: 如果真实工作区在预览和应用之间发生变化，系统必须阻止应用并提示重新执行，避免覆盖用户改动。
- F11: 对于无法安全预览的命令，系统必须阻止执行或降级为明确提示“无法生成 diff，未执行”，不能静默直接修改真实工作区。
- F12: diff 审阅界面必须支持批准和拒绝两个明确动作。
- F13: diff 审阅结果不得泄露被权限系统禁止读取的路径内容。

## 非功能需求
- N1: 默认实现必须保护真实工作区，预览阶段不得直接修改真实文件。
- N2: diff 生成和展示应限制最大文件大小、最大 diff 行数和最大总输出，避免界面卡死。
- N3: 路径处理必须防止预览环境中的变更越界应用到真实工作区之外。
- N4: 审阅流程应复用现有 Textual 内联交互风格，避免新增独立窗口依赖。
- N5: 实现应优先使用结构化文件快照和标准 diff 生成，不依赖脆弱的字符串拼接判断命令副作用。
- N6: 对二进制文件、大文件和编码异常文件，应给出可观测摘要，不强行渲染完整内容。
- N7: 现有权限模式仍然有效；diff 审阅是写入前的额外安全层，而不是完全替代权限检查。
- N8: 新增测试应覆盖文件工具、Bash 预览、批准、拒绝、真实文件变化冲突和命令失败场景。

## 不做的事
- 不承诺回滚网络请求、进程启动、环境变量修改、系统服务修改等非文件副作用。
- 不支持审阅工作区之外的任意系统路径变更。
- 不展示二进制文件的完整内容。
- 不实现复杂三方 merge；真实文件发生变化时直接阻止应用并要求重试。
- 不持久保存每次 diff 审阅内容，除非现有 session/tool 日志自然记录摘要。
- 不改变用户明确启用的 Plan Mode 文档审批流程。

## 验收标准
- AC1: `EditFile` 修改文件时，批准前真实文件不变；批准后文件内容变为 diff 中的新内容。
- AC2: `WriteFile` 创建文件时，批准前文件不存在；批准后文件存在且内容与 diff 一致。
- AC3: `WriteFile` 覆盖文件时，用户能看到旧内容到新内容的 diff。
- AC4: `Bash` 命令修改、创建或删除工作区文件时，系统先展示预览 diff，批准前真实工作区不变。
- AC5: 用户拒绝 diff 后，`EditFile`、`WriteFile`、`Bash` 都不会改变真实工作区。
- AC6: 预览执行失败时，真实工作区不变，并返回失败说明。
- AC7: 预览后真实文件被外部修改时，应用被阻止，并提示工作区已变化。
- AC8: 二进制文件变化显示摘要，不输出不可读内容。
- AC9: diff 审阅界面包含文件摘要、具体 diff、批准和拒绝动作。
- AC10: 相关局部测试和全量测试通过。
